PATCH FRICTION · FOUNDATION3 Jul

Convex-only coupled friction now has a graph-safe state and projection core

The Box3D-inspired prototype keeps every point normal, but will replace eligible per-point tangents with one central, coupled 2D friction block. This is not in the production solve yet; integration and physical gates remain before any performance claim.

  • Eligibility is deliberately conservative: primitives and convex meshes qualify only for unmerged shape-pair columns. Raw meshes, heightfields, cloth/soft contacts, and compound body-pair columns retain existing point friction.
  • The projection uses the full 2×2 tangent effective mass, including its rotational cross term, and the existing PhoenX static/kinetic Coulomb disk law.
  • Warm-start state is stored as a world-space patch impulse. Transfer follows contact matching plus the prior contact-to-column map, so reordered columns or a changed first manifold contact cannot attach history to the wrong pair.
  • Six focused CUDA-graph tests pass: convex/mesh classification, conservative compound fallback, scoped history copy, reordered-column matching, full coupled solve, and kinetic projection.
  • Next: integrate the maximal rigid prepare/iterate path as a compile-time specialization, then require momentum, stacking, static/kinetic friction, mesh fallback, determinism, single-/multi-world, and G1 throughput gates.
BOX3D TRANSFER STUDY3 Jul

Patch friction is promising; two simpler transfers were rejected

The local Box3D source at commit 8441b4a confirms that PhoenX already shares its Soft-Step/TGS core. CPU work stealing and fixed 24-color scheduling do not transfer to the GPU design.

  • Skipping friction in the biased pass fails PhoenX static-friction and lateral force gates because PhoenX permits zero velocity iterations; the prototype was fully reverted.
  • Box3D-like contact softness at ω·h=π/4 is slightly worse than the hard-Hertz π baseline on matched G1 exploration traces; it was discarded.
  • Patch-central friction could remove 31% of scalar contact-row work on G1, 50% on H1, and 44% on DR Legs in measured snapshots. Every measured multi-point column had coherent normals.
  • The prototype will only treat a column as one patch when both shapes are provably convex. Any raw mesh or heightfield pair keeps per-point friction because contact reduction does not preserve disconnected patch identity.

Mission and non-negotiables

  • Main quest: maximize end-to-end PhoenX G1 RL training throughput while keeping repeatable successful training. Faster unsuccessful training does not count.
  • Optimize the reduced-coordinate path where training currently succeeds; improving maximal-coordinate training remains valuable, but do not abandon the proven walker.
  • Keep physics clean and physically grounded: conserve momentum, preserve reliable convergence, retain the hard-Hertz contact model where possible, and use no scene-specific hacks.
  • End-to-end CUDA graph capture is mandatory. Classic PhoenX must remain deterministic.
  • Never overfit to G1. Screen Anymal, H1/G1-class robots, contact-rich scenes, closed loops, mixed ordinary joints/constraints, multiple articulations per world, and both single- and multi-world execution.

How to continue

  • Be innovative. New algorithms and data layouts are welcome; the fastest work is work safely eliminated. Megakernels, Warp tile operations, compact shared memory, and reasonably sized wp.func_native helpers are allowed when evidence supports them.
  • Use Nsight Systems to rank end-to-end costs. Ask for Nsight Compute only when counters can change a decision; prepare one bounded script measuring one kernel, then place its single command in a prominent action box on this page.
  • Move quickly with focused mini-benchmarks and CUDA-graph regression tests. Run full from-scratch training and two robust gates only for meaningful survivors.
  • Compare against nanoG1, MJWarp, FeatherPGS, and full-coordinate PhoenX without trading away accuracy or reliability.
  • Commit measured gains when worthwhile. Keep this page compact, newest information first, remove stale entries, and open it in Chrome after every update.
  • Scope: prioritize PhoenX solver throughput; collision-pipeline gains are allowed when general and quality-preserving. The learner is currently out of scope. This page is the primary communication channel, including privileged-run requests in the top action box.
  • Immediate next step: use the refreshed training ranking to select the next exact, general solver optimization. Do not retry contact-delta register forwarding, wider ABA tiles, final acceleration/twist shuffles, or the full shared response slab.
