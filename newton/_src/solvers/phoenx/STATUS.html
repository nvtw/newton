Task semantics are split cleanly into direct-torque locomotion (Ant/Humanoid) and command-tracking position control (legged robots), while rollout and PPO infrastructure stays shared.

  • Go2: two-world multi-step CUDA graph capture passes with finite state and exact action-to-joint mapping. Graph-leapfrog PPO stays finite at four 5 ms substeps and eight solver iterations.
  • Humanoid: the source task's 21 geared torque actions, 75-value observation, reward/termination equations, implicit actuator gains, self-collision, and 60 Hz control are implemented. Initial reward is exactly 2.6 as specified.
  • Humanoid uses reduced articulation mode by default so every multi-axis motor armature remains physically representable. Its full rollout and PPO update execute in one CUDA graph with a device seed counter; a two-iteration train smoke passes.
  • Ant is now a public reusable environment and its full rollout/update defaults to CUDA graph replay. Promotion exposed and fixed a duplicate up-axis conversion that had made every nominal reset immediately terminal; the regression now checks projected gravity explicitly.
  • H1 now matches the flat-task contact contract: measured ankle-foot colliders replace accidentally omitted mesh contacts; spawn height, biped air-time, foot slide, ankle soft limits, and graph-safe reset command sampling match the source task. A 500-iteration randomized-command curve improves reward −2.34 → +0.47, termination 1.08% → 0.11%, and tracking 0.053 → 0.214 at 133k samples/s.
  • DR Legs hold and walk now use maximal-coordinate PhoenX for all 36 revolute loop constraints, 12 implicit PD actions, two-hop self-collision filtering, graph-captured joint-state reconstruction, and the source 1 ms temporal schedule. After 20 policy steps the maximum loop-anchor residual is 0.392 mm.
  • DR walking now includes the source 1.0 s gait, contact matching, swing clearance, foot parallel/flat rewards, touchdown-speed penalty, and graph-safe reset command distribution. In 100 iterations, hold success improves 0.827 → 0.865 and walking tracking 0.164 → 0.212; walking terminations fall 2.46% → 0.29% at 59k samples/s.
  • CUDA-graph regressions require one-second stable bilateral H1 contact, measured foot collider ownership, DR closed-loop residual below 1 mm, exact action mapping, and graph-replay command sampling. Ant, Humanoid, Go2, and ANYmal-C retain graph smokes, but equivalent long deterministic policy evaluations remain future work—not silently claimed here.
  • Next: pivot back to general PhoenX solver throughput. The Box3D comparison suggests finishing restitution, testing dimensionless contact softness, and prototyping patch-central friction; CPU scheduling and fixed 24-color choices do not transfer directly to this GPU solver.

RETAINED · CROSS-SCENE ROW CACHE3 Jul

Shared articulation factors remove the row builder's worst latency

The existing contact-row Nsight report showed a 943.6 µs kernel with 76.7% of issue spacing in long-scoreboard stalls and 73% excessive global sectors. Every one of 96 row threads repeatedly loaded the same motion subspaces, articulated-inertia columns, and inverse joint factors.

  • The row block now cooperatively loads those exact factors into about 4.6 KB of shared memory once, then evaluates the unchanged articulated response equations and deterministic row order.
  • Matched G1 graph-leapfrog training improves 913.8k → 923.0k samples/s (+1.0%). The retained nanoG1 gap is now about 1.38×.
  • Cross-scene gates also improve: 512-world ANYmal +0.5%, H1 +1.7%; 64 articulations in one world ANYmal +1.5%, H1 +1.7%. Final state magnitudes are identical.
  • Six focused CUDA regressions pass: exact generalized rows, arbitrary contact pages, persistent/reference parity, multiple grounded articulations per world, deterministic dense contacts, and spatial momentum conservation.
  • Commit 2ce2656a is retained on twidmer/experiment-phoenx-persistent-articulations. Repository-wide pre-commit remains blocked only by unrelated pre-existing lint/typo failures; focused lint and formatting pass.
  • Next: use the same profiler evidence to coalesce Jacobian/response writes and shorten path-metadata dependency chains. No user profiling action is required yet.
REJECTED · RESIDENT WORK QUEUE3 Jul

Persistent scheduling recovered the failed fusion, but has not won yet

The retained result remains about 911k samples/s. A profiler-guided resident kernel now uses dynamic full-warp contact tasks, a bounded grid phase barrier, and the existing 8/16/32-thread articulation tiles for dynamics and publication.

  • The original exact contact/ABA/publish fusion reached only about 846k/s. Local 128-thread grouping made contact imbalance worse and fell to 731k/s; that form is rejected.
  • The resident queue removes local group coupling and replaces block-scoped tile operations with an exact scalar warp contact solve. Overflow-page CUDA-graph parity passes.
  • The new kernel uses 104 registers, no spills, 72 bytes of stack, and 2 KB shared memory, versus 122 registers for the original monolith.
  • G1 reaches 900.7k/s: a large recovery over failed fusion, but still 1.4% below the retained 913.8k/s matched baseline. The scalar warp solve alone reaches 909.0k/s, so it is not independently a win.
  • Moving row construction into that monolith fell further to 890.7k/s: an extra grid barrier and instruction footprint outweighed the removed launch. The queue remains useful architectural evidence, not an active path.
  • No new Nsight Compute action is required from the user yet. Acceptance still requires G1, ANYmal, H1, single-/multi-world, determinism, and arbitrary-contact graph gates.
REJECTED · MATRIX-FREE JACOBIAN3 Jul

The monolith is latency-bound, but path reconstruction overfits G1

The requested one-launch Nsight Compute report rules out further blind fusion and points toward compact, phase-specific warp cooperation: preserve residency only where it deletes traffic without lengthening dependency chains.

  • The exact fused biased-contact/ABA/publish kernel takes 1.42 ms, uses 122 registers/thread, and reaches only 30.3% achieved occupancy. DRAM reaches 18.7% of peak, so raw bandwidth is not the limiter.
  • Schedulers have an eligible warp only 40.8% of cycles. Long-scoreboard dependencies consume 3.60 of the 9.01 cycles between issued instructions (39.9%); average active width is 15.1 threads/warp and predication reduces it to 13.0.
  • Global accesses waste 46% of transferred sectors; L1/L2 hit rates are already 75.0%/75.6%. The useful target is coalesced structure-of-arrays access and shorter dependency chains, not __ldg() or cache hints.
  • An exact matrix-free solve removed the global Jacobian matrix and passed overflow-page CUDA-graph parity. Matched G1 improved 913.8k → 922.8k/s (+0.98%).
  • The required cross-scene gates reject it: 512-world ANYmal/H1 fall 730.4k → 705.3k/s (−3.4%) and 708.9k → 687.9k/s (−3.0%); 64 articulations in one world fall 89.1k → 85.3k/s (−4.2%) and 90.9k → 89.3k/s (−1.8%). States match exactly.
  • The experiment is preserved on twidmer/experiment-phoenx-matrix-free-contact-rows at 030be01c. Next: redesign contact response construction around coalesced warp tiles and shorter load-use chains, without recomputing sparse Jacobians or absorbing the full ABA/publish phase.
REJECTED · DEVICE-CALL PHASE SPLIT3 Jul

A real compiler boundary makes the megakernel 24% slower

A small general Warp prototype added explicit no-inline device functions. PTX proves that post-contact ABA survives as a separate .func/call.uni phase, rather than being flattened into the contact kernel.

  • The existing overflow-page CUDA-graph parity regression passes, so the split does not alter the equations, deterministic schedule, or arbitrary-contact behavior.
  • Same-session G1 graph-leapfrog training measures about 700k/s for the no-inline megakernel versus 921k/s for the reference path. The former inline megakernel was about 846k/s.
  • The large device-call signature forces parameter marshalling and local-stack traffic; that cost is substantially worse than the register-lifetime pressure it removes.
  • Static assembly reports 122 registers, no spills, a 72-byte stack frame, and 18,696 SASS instructions. Separate contact and advance/publish phases use 80 and 101 registers respectively; fusion duplicates almost no instructions but raises the peak register footprint.
  • A meaningful 20-block residency target caps the kernel to 96 registers, but introduces 156/100 bytes of spill stores/loads and falls to about 834k/s versus roughly 846k/s unbounded. A four-block hint changes no resources and reaches about 837k/s.
  • Decision: the completed full-counter capture selects a compact-context, coalesced phase design; further enlargement of this monolith is rejected.
RETAINED EXPERIMENT · CROSS-ROBOT WIN3 Jul

Contact relax and final publication now execute as one articulation phase

The persistent path now executes contact velocity refresh, bias-free generalized contact solve, contact-delta application, and final articulation publication in one graph-captured kernel. The known-regressing factor fusion was removed from the active composition.

  • Full graph-leapfrog G1 training measures about 911.2k samples/s persistent versus 901.1k/s reference in the matched bracket (+1.1%). Saturated standalone G1 physics is effectively neutral/slightly negative (−0.35%), confirming that end-to-end overlap must remain the primary metric.
  • Contact-rich 512-world fleets improve ANYmal 365.6k → 381.0k/s (+4.2%) and H1 343.2k → 362.7k/s (+5.7%).
  • Many articulations in one world also improve: 64-robot ANYmal 45.8k → 47.1k/s (+2.9%) and H1 46.1k → 47.7k/s (+3.5%). Final state magnitudes match reference in every bracket.
  • A CUDA-graph regression covers two substeps and more than two contact pages. Generalized state, body pose/twist, impulses, manifold data, and derived contact rows match the reference within 2e−6.
  • The implementation supports the arbitrary-contact overflow loop and exposes no user contact-limit tuning. The reference execution path remains unchanged and selectable.
  • Next: preserve this bounded fusion and pursue phase elimination only when the replacement reduces both global traffic and live working set.
REJECTED · EXACT RESPONSE BASIS3 Jul

Adaptive contacted-link bases lose end-to-end throughput

An exact experimental path classified articulations touching at most two links, built six unit-wrench articulated responses for those links, and synthesized every contact row from that compact basis. Arbitrary-contact and denser articulations retained the direct reference path.

  • Full graph-leapfrog G1 training measures about 898k/s versus the retained 911k/s (−1.5%). Contact-rich ANYmal measures 374.9k/s versus 381.0k/s (−1.6%); H1 measures 360.9k/s versus 362.7k/s (−0.5%).
  • The algebra is exact, but float reconstruction introduces roughly 1e−3 response stress error, 5e−5 state error, and up to 0.2% derived effective-mass error. Those bounded differences do not buy performance.
  • Only about 777 of 8,192 sampled G1 worlds qualified at runtime. Basis construction, synchronization, and coefficient traffic overwhelm the rows it replaces; this is not a general cross-scene win.
  • The prototype is preserved on twidmer/experiment-phoenx-contact-response-basis at cdabaca7 and reverted from the active path at 5b8677e5.

Shared-resident contact rows are exact, but slower

A separate graph-static articulation path cached 32 Jacobian and articulated-response rows in shared memory and routed denser articulations through the reference solver. This tested a substantially different memory strategy without risking the retained implementation.

REJECTED · WORKING-SET OVERFUSION3 Jul

Biased contact solve plus ABA is exact, but 7–9% slower

A larger megakernel continued from biased generalized contact solving through Coriolis ABA, momentum capture/restore, and intermediate publication. Overflow-page CUDA-graph parity remained within 2e−6, proving the reordered schedule was physically equivalent for topology-proven static contacts.

  • Saturated G1 physics fell from about 1.91M to 1.74M steps/s (−9%). Full graph-leapfrog training fell from the retained 911k/s to about 846k/s (−7%).
  • The contact tile and ABA traversal share almost no live working set; combining them increases register pressure more than it removes memory traffic.
  • The exact prototype is preserved on twidmer/experiment-phoenx-biased-contact-megakernel at 6777ae00 and reverted from the active path at c13deef5.
  • The replacement direction is an exact low-rank contacted-link response basis. Deterministic runtime classification is now graph-captured and covered by the arbitrary-contact regression; no robot-name specialization or user tuning is involved.

Next decisions

Return to end-to-end work elimination

The response-basis study shows that optimizing row construction in isolation is not enough: extra operator construction, synchronization, and overlap pressure can dominate. The retained sparse-row path remains the best measured exact implementation.

Next: consume the bounded full-counter capture of the exact inline contact/ABA/publish kernel. Keep the large fusion only if the counters support a concrete compact-context or instruction-footprint redesign; otherwise return to eliminating a complete standalone phase with the retained bounded fusion as the safe baseline.

Do not repeat

  • Separate 32-row resident Jacobian/response solve: bit-exact but −8.6% G1 training from a second launch, fixed shared loads, and occupancy.
    • One-pass ABA / pre-contact-only Coriolis: +6% physics but severe momentum drift.
    • Compile-time velocity-only publish kernel: exact but +0.1% (neutral).
    • Factor/contact multi-stream overlap: −2.7% training from GPU contention.
    • Conditional empty-reset FK/observation: neutral in overlapped training and −1.7% in high-reset stepping; fully reverted.
    • Always-transposed response rows: faster physics, −2.7% training.
    • Early response-vector prefetch: +0.55% physics but −0.64% training from increased live-register pressure.
    • Drop tiled ABA generalized_rhs store: exact but about −0.6% physics; fully reverted.
    • Serial three-row contact scheduling: 1.06M versus 1.93M physics steps/s; row parallelism is essential.
    • Row blocks of 32 gain 0.56% isolated physics but lose 0.38% training. Blocks of 64 gain 0.65% G1 training overlap but lose 1.2–2.7% on contact-rich G1/H1/ANYmal fleets; retain the general 96-thread block.
    • Stride-3 contact prepare: policy exploits stale-contact physics and fails the true-physics gate.
    • Factor sub-warp tiling: +9% kernel time.
    • Compile-time generalized-contact mode specialization: exact focused tests pass, but static iteration expansion drops G1 physics about 3%; keep the compact runtime loop.
    • Fuse per-joint initialization into tiled factor: physically exact, but −2.3% G1 physics from increased kernel pressure.
    • Path-sparse Jacobian cache: −1.5% physics; indexed tile dots slowed every PGS update.
    • Remove ABA generalized-acceleration stores: neutral/slightly slower.
    • Full 96-row shared response cache: −14% from fixed loads and occupancy.
    • Adaptive two-link unit-wrench response basis: exact algebra, but −0.5–1.6% across G1, ANYmal, and H1 training/robot brackets.
    • No-inline ABA phase inside the contact megakernel: exact and graph-safe, but about −24% G1 training from device-call marshalling/local-stack traffic.
    • Launch bounds on the inline megakernel: four blocks leave its 122 registers unchanged; 20 blocks cap it to 96 with spills and regress training to about 834k/s.
    • Shared ABA-factor cache in contact-row blocks: mixed by articulation width/batch and −3–5% on smaller G1/ANYmal fleets; moving global reads to a synchronized shared phase is not a universal latency fix.
    • Row-builder local ancestor stack: −25%.
    • BF16 recurrent forward: +0.3%, removed.

    Mission and non-negotiables

    • Main quest: maximize end-to-end PhoenX G1 RL training throughput while keeping repeatable successful training. Faster unsuccessful training does not count.
    • Optimize the reduced-coordinate path where training currently succeeds; improving maximal-coordinate training remains valuable, but do not abandon the proven walker.
    • Keep physics clean and physically grounded: conserve momentum, preserve reliable convergence, retain the hard-Hertz contact model where possible, and use no scene-specific hacks.
    • End-to-end CUDA graph capture is mandatory. Classic PhoenX must remain deterministic.
    • Never overfit to G1. Screen Anymal, H1/G1-class robots, contact-rich scenes, closed loops, mixed ordinary joints/constraints, multiple articulations per world, and both single- and multi-world execution.

    How to continue

    • Be innovative. New algorithms and data layouts are welcome; the fastest work is work safely eliminated. Megakernels, Warp tile operations, compact shared memory, and reasonably sized wp.func_native helpers are allowed when evidence supports them.
    • Use Nsight Systems to rank end-to-end costs. Ask for Nsight Compute only when counters can change a decision; prepare one bounded script measuring one kernel, then place its single command in a prominent action box on this page.
    • Move quickly with focused mini-benchmarks and CUDA-graph regression tests. Run full from-scratch training and two robust gates only for meaningful survivors.
    • Compare against nanoG1, MJWarp, FeatherPGS, and full-coordinate PhoenX without trading away accuracy or reliability.
    • Commit measured gains when worthwhile. Keep this page compact, newest information first, remove stale entries, and open it in Chrome after every update.
    • Scope: prioritize PhoenX solver throughput; collision-pipeline gains are allowed when general and quality-preserving. The learner is currently out of scope. This page is the primary communication channel, including privileged-run requests in the top action box.
    • Immediate next step: prototype cross-phase work elimination or bounded persistent articulation execution; gate it on full training early. Do not retry explicit response-basis operators, contact-delta register forwarding, wider ABA tiles, final acceleration/twist shuffles, or the full shared response slab.
