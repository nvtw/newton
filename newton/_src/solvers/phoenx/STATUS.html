Training is now the performance benchmark; safe tuning helps, but no large solver win is proven yet

A complete Nsight Systems trace covers a real PPO iteration, and every candidate is measured end to end with the G1 training workload rather than an isolated microbenchmark.

  • The best measured full-coordinate baseline uses 8192 worlds and sustains about 202k samples/s. The pinned nanoG1 result is 1.276M samples/s, so the remaining gap is about 6.3×.
  • 8192 worlds beats 4096 and 16384. A 32-thread block per world beats 64/128-thread blocks and the fast-tail scheduler by 27–29% on this scene.
  • BF16 and reducing PGS work produce only a small additional throughput gain. More aggressive reductions are not accepted because the full-coordinate policy still misses the frozen quality gate.
  • Nsight attributes 19.3% of GPU time to block-world prepare/PGS, 6.6% to multi-world coloring, about 6.0% to physics radix sorts, and 3.6% to narrow phase. Neural-network forward and gradient kernels account for roughly 41%.
  • The next code experiments target reusable/static joint topology, cheaper dynamic-contact coloring/sorting, and solve-kernel traffic. Each must preserve CUDA graph capture, determinism, physical regressions, and frozen-policy quality.

Verdict: profiling and training-based A/B measurement are in place; configuration tuning is proven, but a major general-purpose PhoenX optimization has not yet landed.

ARMATURE VALIDATED30 Jun · latest

Full-coordinate PhoenX now uses physical reflected rotor inertia

The unstable auxiliary armature constraint row has been removed entirely. Rotational motor armature is instead reflected through the gearbox and added as a rank-one inertia tensor to the rotor-side child body, in the child body frame. The authored parent inertia contains the stator/link inertia.

  • The previously failing full 29-joint G1 free-space CUDA-graph regression now passes instead of diverging beyond 6 rad.
  • Analytical pendulum tests pass across an armature sweep, and gear tests pass the expected gear² × rotor inertia period scaling, including a 100:1 ratio.
  • Off-axis tensor placement, free-floating angular-momentum conservation, and live domain-randomization updates have direct regressions; zero armature remains a no-op.
  • The model is deterministic and uses ordinary body inertia and gyroscopic dynamics—no relaxation, interaction-count scaling, or solver tuning.
  • Scope is deliberately physical: maximal mode currently accepts rotational armature on revolute joints. Translational and multi-axis armature are rejected instead of approximated inaccurately.

Current verdict: validated for the revolute robot-motor case. All 18 focused CUDA-graph tests pass, including the formerly divergent G1 chain. Removing the obsolete row also removes two fields, its solve branches, and two per-body substep velocity snapshots from the full-coordinate hot path. Next: prove successful maximal-coordinate G1 policy training, then optimize measured throughput without weakening the gate.

ROOT CAUSE ISOLATED30 Jun

Pure-maximal G1 armature is not yet reliable

A controlled free-space, zero-action sweep isolates the failure to the maximal-coordinate armature path rather than contacts, friction, policy actions, or an insufficient iteration count.

  • With armature disabled, the full 29-joint G1 remains at its home pose to numerical precision over the test horizon.
  • With physical armature enabled, the same chain diverges. Raising the schedule as far as 10 substeps × 64 PGS iterations does not fix it and convergence is non-monotonic.
  • The current pendulum and three-link armature tests are therefore insufficient. The fix must add a deep spatial-chain, CUDA-graph regression and preserve momentum and the exact configured armature.
  • The equal-work substep-versus-iteration claim cannot be judged honestly until this defect is removed.
  • An exact local 2×2 drive/armature block did not improve the failure, proving the problematic coupling spans the tree joint-lock rows rather than just the motor row.
  • Armature-row relaxation at 0.1 passes the G1 stability check but destroys physical accuracy: pendulum period errors reach 67% and three-link modal errors reach 47%. It is rejected, not retained.
  • The clean next implementation is an exact tree-coupled maximal mass operator for M_body + Jᵀ A J (tree LDL/Woodbury), while keeping maximal joint and contact rows. This has the correct fixed-time-step physics and avoids tuning parameters.

Existing Nsight traces also identify the first performance targets. In classic contact-rich G1, radix sorting consumes 21.1% of GPU kernel time, block-world prepare/iterate 20.8%, greedy coloring 8.0%, and scans 5.9%. In the articulated contact-rich path, generalized contact construction/solve/gather/finalize dominates roughly 55% before factorization and momentum passes. Correctness remains first; these measurements define the subsequent optimization work.

NOT YET COMPETITIVE30 Jun · 22:20

Hybrid versus FeatherPGS

There is no valid same-scene, same-GPU result proving hybrid competitive with FeatherPGS. Claiming otherwise would be misleading.

  • FeatherPGS is articulation-specialized and designed around high-throughput parallel articulation solves.
  • Hybrid currently pays for both ABA response and retained maximal rows, so it likely loses raw articulation throughput.
  • Hybrid has demonstrated stable integration with general PhoenX contacts and constraints; it has not demonstrated world-class speed.
  • A direct Anymal/G1 benchmark against FeatherPGS is now an explicit performance requirement, after the pure-maximal schedule/stability sweep.

HYBRID EXPLAINED30 Jun · 22:08

Is hybrid the perfect full/reduced-coordinate solution?

No, but it is a clean and useful middle ground. Hybrid keeps every full-coordinate PhoenX joint, loop, limit, motor, and contact row in deterministic PGS. Instead of applying each row through independent rigid-body inverse masses, it applies impulses through an exact linear-time articulated-body response for each tree.

  • What it gains: exact tree inertia coupling and armature response, much better conditioning for stiff humanoid drives, momentum conservation, and reduced-like robot stability while retaining maximal PGS constraints and contacts.
  • What remains maximal: loop closures, contacts, friction, limits, and joint-row residuals are still iterated by PGS. Their convergence and accuracy are not automatically identical to a direct reduced-coordinate solve.
  • Applicability: it supports multiple articulations per world and many worlds, plus ordinary contacts and maximal constraints. The accelerated path currently requires standard PhoenX, the fast-tail schedule, and no mass splitting; unsupported scene features need broader engineering.
  • Performance: it is not free. Hybrid evaluates articulated dynamics while also retaining maximal rows, so it cannot honestly be called full speed of the pure full-coordinate solver. Earlier matched measurements showed meaningful overhead; Nsight profiling is still pending.
  • Verdict: excellent for stiff tree robots embedded in contact-rich maximal scenes, but not yet a universal final solver. A world-class endpoint would preserve its conditioning while eliminating redundant work and proving quality/performance across loops, many articulations, dense contacts, and diverse robots.
